DevOps의 전신인 애자일 개발 방법론과 SRE, ChatOps에 대한 이론 강의

1. 애자일 개발
   등장 배경

폭포수(Waterfall) 모델의 한계

치밀한 시장 조사 → 기획 → 개발 → 납기 내 릴리즈
완성했지만 경쟁사에 시장 선점 당해 사업 실패

"계획대로 릴리즈(X) + 비즈니스 가치 제공(O)"

핵심 개념
개념설명Iteration1~4주 단위 짧은 개발 주기. 계획 → 개발 → 피드백 반복User Story"무엇을 달성하고 싶다"는 요구사항을 문장으로 표현Epic광범위한 요구사항. User Story로 세분화(Breakdown)
Scrum 개발
럭비의 Scrum에서 유래한 애자일 방법론
3가지 역할(Role):

Product Owner: 상품 가치 극대화 책임. Product Backlog 우선순위 관리
Developer: 개별 기능 개발. 3~9명 적정. 자율적 자기 조직
Scrum Master: 팀 최고 책임자. 개발 방해 요인 제거

Sprint 흐름:
릴리즈 계획 → 스프린트 계획 → 스프린트(개발) → 일간 스크럼 → 스프린트 리뷰 → 스프린트 회고
일간 스크럼 (15~30분):

어제 한 일
오늘 할 일
Blocker (진행 방해 요소)

스프린트 회고:

좋았던 것
나빴던 것
다음 Sprint에서 반드시 해야 할 일

핵심 철학

"Do not just Do Agile, Be Agile"
도구 도입에 만족하지 말고, 변화에 맞서 계속하는 것

2. SRE (Site Reliability Engineering)
   개요

Google의 운영 노하우에서 탄생
DevOps를 운용하는 방법의 하나
DevOps와 달리 운용에 초점을 맞춘 개념

SRE 팀의 미션
가용성 / 지연 / 성능 / 효율화 / 변화 관리 / 모니터링 / 긴급 대응 / 용량 관리
SRE 접근법
영역내용시스템 튜닝하드웨어 성능 문제, 애플리케이션 구현 문제 분석모니터링서비스 모니터링(현재) + Capacity 모니터링(미래 예측)작업 품질자동화/간소화로 변경 관리장애 대응단시간에, 적은 인원으로 신속 해결
DevOps vs SRE
DevOps → 안정적 가동 + 최적화 → 비즈니스 실현
SRE → 사이트 신뢰성 향상 → 비즈니스 실현

목적은 동일! 접근 방식이 다를 뿐

3. ChatOps
   개요

운용에 관련된 다양한 작업을 채팅 도구를 통해 효율화
Slack 등 채팅 도구 + Bot + 시스템 연동

구조
ITS / CI도구 / 모니터링
↓
Bot
↓
채팅 도구 (Slack 등)
↙ ↘
개발 부문 운용 부문
ChatOps 장점

커뮤니케이션 도구 일원화
작업 ShortCut (복잡한 작업을 채팅 타이핑으로 실행)
조작 및 결과를 팀 전원이 공유
이벤트와 커뮤니케이션을 하나의 타임라인으로 통합

ChatOps 자동화 가능 항목

- 애플리케이션 Build
- 테스트 실행
- 애플리케이션 Deploy
- 시스템 Resource 현황 표시
- Alert 통지
